# 路线图

[英文](./roadmap.md) | 简体中文

Sites Relay 专注于固定上游的应用层流量。本路线图记录产品方向和决策边界；可执行任务放入 GitHub Issues。

## 产品原则

- 所有目标均锁定为服务端配置。
- 在暴露路由策略或注入上游凭据之前完成鉴权。
- 对配置不完整、不支持的内容、重定向、超限流量和超时采用带明确错误的 fail-closed 处理。
- 保持小而可审计的中继契约。
- 将完整浏览器兼容放入使用独立 origin 的远程浏览器架构。

## 当前版本线：v0.1

v0.1 用于建立接近生产形态的参考实现：

- 固定上游 JSON/SSE 中继，包含方法、路径、查询、请求头、大小和超时策略
- 面向程序化客户端的令牌鉴权
- 面向同源浏览器应用、使用精确邮箱白名单的 Sites 用户鉴权
- 可选的身份保护静态网页镜像，包含 HTML 净化和 CSS 重写
- 不含密钥的健康诊断、上游 hostname 隐私、回归测试和 CI
- 成对的中英文文档、部署示例和仓库内置 Codex Skill

当前仍属于 1.0 之前的版本。配置名和响应契约调整需要提供足以支持迁移工作的安全性或可用性收益。

## 近期优先事项

1. 使用服务端运行时密钥和真实固定 JSON/SSE 上游验证文档中的部署流程。
2. 添加能够展示不同策略或流式模式、与供应商无关的配置范例。
3. 保持诊断无密钥且真实：`ready` 报告静态验证，实时请求报告上游连通性。
4. 使用具体部署反馈指导路由与持久化工作。

## 评估中的方向

以下评估方向将在获得真实部署证据后推进：

- 多个命名的服务端固定上游，每个上游使用独立路由策略
- 抽取策略与流量限制核心，以支持更多部署适配器
- 使用持久化存储实现可选聚合请求计数，并排除正文、凭据和用户身份
- 保持默认拒绝模型的可复用配置预设

## 独立 Browser Relay 方向

完整浏览器会话采用远程浏览器隔离、逐用户会话、导航策略、滥用控制和独立 origin。专用架构负责这项能力，`/api/proxy/*` 与 `/web/*` 保持当前契约。详见[完整网页兼容方向](./web-compatibility-direction.zh-CN.md)。

## 扩展门槛

路线图准入条件包括具体 Sites 用例、服务端控制的目标、可测试安全边界，以及多个真实部署提出的需求。早期想法继续作为示例、适配器或独立项目推进。
